# Документация информационной системы «PartStock»

## 1. Назначение и область применения

Информационная система «PartStock» предназначена для подбора и заказа автозапчастей с учётом совместимости с конкретным автомобилем.  
Система ориентирована на:

- частных автовладельцев;
- небольшие сервисы и СТО;
- магазины автозапчастей, работающие в онлайне.

Основные задачи системы:

- предоставить удобный поиск и фильтрацию автозапчастей;
- обеспечить подбор по VIN и по параметрам автомобиля;
- позволить пользователям формировать корзину и оформлять заказы;
- дать продавцам и администраторам инструмент для работы с каталогом (на этапах дальнейшей разработки).

## 2. Общая характеристика системы

### 2.1 Основные роли

- **Гость** — может просматривать каталог, карточки товаров, подбирать по VIN.
- **Авторизованный пользователь** — дополнительно:
  - имеет личный кабинет (профиль);
  - может сохранять свои данные для оформления заказов (на следующих этапах);
  - может формировать корзину и переходить к оформлению.

### 2.2 Основные функции

- просмотр каталога товаров по категориям;
- просмотр карточки конкретного товара;
- поиск по названию, артикулу и OEM-номеру;
- подбор по VIN (демонстрационный режим на текущей версии);
- управление корзиной (добавление, изменение количества, удаление);
- регистрация и вход в систему;
- отображение профиля пользователя в личном кабинете.

## 3. Архитектура и технологический стек

### 3.1 Архитектура

Система реализована по классической трёхзвенной архитектуре:

- **Клиент (Frontend)** — одностраничное приложение (SPA) на React.
- **Сервер приложений (Backend)** — REST‑API на Flask.
- **База данных** — SQLite (демонстрационная среда).

Взаимодействие:

- клиент обращается к серверу по HTTP (JSON‑API);
- сервер работает с БД через ORM (SQLAlchemy).

### 3.2 Технологический стек

- **Frontend**
  - TypeScript
  - React
  - React Router
  - Mantine UI
  - Vite

- **Backend**
  - Python
  - Flask
  - Flask‑CORS
  - Flask‑SQLAlchemy

- **Базы данных**
  - SQLite (файл `partstock.db`)

- **Инфраструктура**
  - npm (управление зависимостями фронтенда)
  - виртуальное окружение Python (рекомендуется) для бэкенда

## 5. Модель данных

### 5.1 Основные сущности

- **User (Пользователь)**
  - `id` — идентификатор;
  - `name` — имя;
  - `email` — почта (уникальная);
  - `password` — пароль (в текущей учебной версии хранится в открытом виде, на бою должен храниться в хеше).

- **Product (Товар)** — на фронтенде используется как тип данных:
  - `id` — идентификатор;
  - `name` — название;
  - `description` — описание;
  - `price` — цена;
  - `image` — URL изображения;
  - `article` — артикул;
  - `oem` — OEM‑номер;
  - `categoryId` — идентификатор категории;
  - `categoryName` — название категории;
  - `compatibility` — список строк с описанием совместимости (марка/модель/поколение).

- **Category (Категория)**
  - `id` — идентификатор;
  - `name` — название категории (например, «Тормозная система»);
  - `slug` — URL‑слаг.

## 6. API сервера

Базовый URL: `http://127.0.0.1:5000`

### 6.1 Категории

**GET `/api/categories`**  
Возвращает список категорий.

Пример ответа:

```json
[
  { "id": "1", "name": "Тормозная система", "slug": "tormoznaya-sistema" },
  { "id": "2", "name": "Подвеска", "slug": "podveska" }
]
```

### 6.2 Товары

**GET `/api/products`**  
Возвращает список товаров.

**GET `/api/products/<id>`**  
Возвращает информацию о конкретном товаре.

Пример ответа:

```json
{
  "id": "1",
  "name": "Тормозные колодки Brembo передние",
  "description": "Передние тормозные колодки Brembo для VW Golf 7, Tiguan. Оригинальное качество.",
  "price": 4200,
  "image": "https://placehold.co/800x600/2f2f2f/ffffff?text=Brembo+Pads",
  "article": "P83123",
  "oem": "5Q0 698 151",
  "categoryId": "1",
  "categoryName": "Тормозная система",
  "compatibility": ["VW Golf VII", "VW Tiguan I"]
}
```

### 6.3 Аутентификация и профиль

**POST `/api/auth/register`**  
Регистрация нового пользователя.

Тело запроса:

```json
{
  "name": "Иван",
  "email": "ivan@example.com",
  "password": "secret"
}
```

Ответ (успех):

```json
{
  "id": 1,
  "name": "Иван",
  "email": "ivan@example.com"
}
```

Ошибки:

- `400 name_email_required` — не указаны имя или email;
- `409 email_taken` — пользователь с таким email уже существует.

**POST `/api/auth/login`**  
Авторизация пользователя.

Тело запроса:

```json
{
  "email": "ivan@example.com",
  "password": "secret"
}
```

Ответ (успех) — аналогичен успешной регистрации.  
Ошибка `401 invalid_credentials` — неправильная пара email/пароль.

**GET `/api/profile/<user_id>`**  
Получение профиля пользователя по идентификатору.

## 7. Интерфейс пользователя

### 7.1 Главная страница

Содержит:

- крупный заголовок и описание сервиса;
- кнопку «Подбор по VIN»;
- блок категорий (карточки категорий);
- блок «Популярные товары» (витрина товаров).

### 7.2 Каталог

Страница `/catalog`:

- список товаров;
- поле поиска по названию, артикулу и OEM;
- фильтрация по категории при переходе по URL `/catalog/:categorySlug`.

### 7.3 Карточка товара

Страница `/product/:id`:

- фото товара;
- название;
- цена;
- артикул и OEM‑номер;
- совместимость (список моделей);
- кнопка «В корзину».

### 7.4 Корзина

Страница `/cart`:

- список выбранных товаров;
- количество, цена за единицу, сумма по строке;
- изменение количества;
- удаление товара из корзины;
- итоговая сумма и кнопка «Оформить заказ» (в текущей версии без интеграции оплаты).

### 7.5 Подбор по VIN

Страница `/vin`:

- поле ввода VIN (17 символов);
- валидация (количество символов, допустимые символы);
- кнопка запуска подбора;
- вывод демонстрационного списка подходящих запчастей (на основе существующих демо‑товаров).

### 7.6 Личный кабинет

Страница `/account`:

- если пользователь не авторизован:
  - вкладки «Вход» и «Регистрация»;
  - формы для ввода email, пароля, имени;
  - отображение ошибок (неверные данные, занятый email).

- если пользователь авторизован:
  - отображается имя и email;
  - есть кнопка «Выйти»;
  - зарезервирован блок под историю заказов и настройки (будущие доработки).

Кнопка перехода в личный кабинет находится в правой части шапки (иконка пользователя, текст «Войти» или имя пользователя).

## 8. Безопасность и ограничения

Текущая учебная версия имеет ряд упрощений:

- пароли хранятся без хеширования;
- отсутствует полноценная сессия или JWT‑токены;
- нет разграничения прав доступа по ролям (админ, продавец);
- оплата и доставка работают только на уровне ТЗ (интеграции не подключены).

При выводе проекта в продакшен необходимо:

- добавить хеширование паролей (например, bcrypt);
- реализовать авторизацию через токены (JWT) или сессионные cookies;
- добавить роли пользователей и проверки прав на уровне API;
- настроить защищённый протокол (HTTPS) и хранение секретов во внешней конфигурации.
