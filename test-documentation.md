# Тестовая документация информационной системы «PartStock»

## 1. Цели и объём тестирования

Цель тестирования — проверить корректность работы основной функциональности учебной версии системы «PartStock»:

- просмотр каталога и карточек товаров;
- подбор запчастей по VIN (демо‑режим);
- управление корзиной;
- регистрация и авторизация пользователя;
- отображение личного кабинета.

Нагрузочное, безопасность на продакшен‑уровне и интеграции с платёжными/логистическими сервисами в рамках данной версии **не тестируются**.

## 2. Объекты тестирования

- Web‑интерфейс:
  - главная страница `/`;
  - каталог `/catalog`, `/catalog/:categorySlug`;
  - карточка товара `/product/:id`;
  - корзина `/cart`;
  - подбор по VIN `/vin`;
  - личный кабинет `/account`.

- REST‑API (Flask):
  - `GET /api/categories`;
  - `GET /api/products`;
  - `GET /api/products/<id>`;
  - `POST /api/auth/register`;
  - `POST /api/auth/login`;
  - `GET /api/profile/<user_id>`.

## 3. Среда тестирования

- ОС: Windows 10 / 11.
- Браузер: Google Chrome / Microsoft Edge (актуальная версия).
- Backend: Flask (запуск через `python ../backend/app.py` из скрипта `npm run dev:full`).
- БД: SQLite, файл `partstock.db`.
- Frontend: Vite dev‑сервер (`npm run dev`), адрес `http://localhost:5173` или `http://localhost:5174`.

## 4. Типы тестирования

- **Функциональное тестирование** пользовательских сценариев.
- **Тестирование пользовательского интерфейса** (наличие элементов, базовая вёрстка).
- **Позитивные и негативные тесты** форм (регистрация, логин, VIN).

## 5. Тестовые сценарии (высокий уровень)

### 5.5 Регистрация и авторизация

| ID        | Название                                   | Шаги                                                                                   | Ожидаемый результат                                                    |
|-----------|--------------------------------------------|----------------------------------------------------------------------------------------|------------------------------------------------------------------------|
| AUTH-01   | Регистрация нового пользователя           | Открыть `/account`, вкладка «Регистрация», ввести валидные имя/email/пароль, нажать   | Пользователь создаётся, отображается личный кабинет                   |
| AUTH-02   | Повторная регистрация с тем же email      | Повторить AUTH-01 с тем же email                                                      | Появляется ошибка о занятости email (`email_taken`)                   |
| AUTH-03   | Вход с корректными данными                | Зарегистрировать пользователя, затем во вкладке «Вход» ввести корректные email/пароль | Вход успешен, показывается личный кабинет                             |
| AUTH-04   | Вход с неверным паролем                   | Во вкладке «Вход» ввести существующий email и неверный пароль                         | Сообщение об ошибке «Неверный email или пароль»                       |
| AUTH-05   | Выход из аккаунта                         | В личном кабинете нажать кнопку «Выйти»                                               | Пользователь считается разлогиненным, кнопка в шапке снова «Войти»   |

## 6. Критерии приёмки

- Все описанные выше сценарии выполняются без критических ошибок;
- не возникает падений фронтенда (белый экран с ошибкой JS в консоли);
- API возвращает корректные коды ответов (2xx при успехе, 4xx при ошибках ввода);
- базовая вёрстка не «ломается» на стандартном разрешении ноутбука/монитора.

## 7. Результаты тестирования

На момент подготовки тестовой документации система находится в учебной разработке, поэтому:

- функциональные сценарии, описанные в разделе 5, являются **целевыми**;
- при фактическом прогоне тестов необходимо зафиксировать:
  - дату прогона;
  - версию кода (коммит или дату);
  - статус каждого теста (Passed / Failed);
  - найденные дефекты (с описанием, шагами воспроизведения и ожидаемым результатом).

Результаты рекомендуется оформлять в отдельной таблице или файле отчёта (например, `docs/test-report.md`).
